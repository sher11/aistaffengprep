Sharding Challenges

  • Cross-shard queries: Joins across shards are expensive
  • Rebalancing: Moving data when adding shards
  • Hotspots: Uneven data distribution

Replication

Replication Patterns ▼
flowchart LR subgraph "Leader-Follower" L1[Leader] --> F1[Follower 1] L1 --> F2[Follower 2] end subgraph "Multi-Leader" L2[Leader 1] <--> L3[Leader 2] end subgraph "Leaderless" N1[Node 1] <--> N2[Node 2] N2 <--> N3[Node 3] N3 <--> N1 end

Leader-Follower (Most Common)

  • All writes go to leader
  • Reads from leader or followers
  • Simple, but leader is bottleneck

Multi-Leader

  • Multiple nodes accept writes
  • Conflict resolution needed
  • Good for multi-datacenter

Leaderless (Dynamo-style)

  • Any node accepts writes
  • Quorum-based consistency
  • High availability

Indexing

Index Types & Trade-offs ▼

B-Tree Index

  • Default for most RDBMS
  • Good for range queries and equality
  • O(log n) lookup

Hash Index

  • O(1) for equality lookups
  • No range query support
  • Used in memory (Redis)

LSM-Tree (Log-Structured Merge)

  • Optimized for writes
  • Used in Cassandra, RocksDB
  • Background compaction
Indexing Trade-off: More indexes = faster reads, slower writes, more storage
